The analogy
Think of full aggregation like a census — IIQ reads every resident each time. Delta aggregation is like a change-of-address notification — IIQ only processes people who moved since the last postman came. The UPDATED_ON column is the postmark date.

1 — The Oracle view (or query) IIQ will use
Delta aggregation in IIQ works by passing a timestamp parameter into the SQL query at runtime. You need to expose this via the existing IDENTITY_AUTH_SOURCE_VW or a dedicated delta view.

The cleanest approach is a parameterized query directly in the application's JDBC config, not a view, because views can't accept parameters.

sql
-- This is what goes into IIQ's "Delta Aggregation Query" field
SELECT
    ROBID,
    FLID,
    REFOG,
    ENTITYID,
    ENTITY,
    MANAGER,
    LASTNAME,
    FIRSTNAME,
    COUNTRY,
    LANGUAGE,
    FUNCTION,
    EMAIL,
    TYPE,
    STATUS,
    ISTEST,
    ISRPA,
    SOURCE,
    UPDATED_ON
FROM OWN_AP12008.STG_MASTER_ROBI
WHERE IS_CURRENT = 'Y'
  AND UPDATED_ON >= ?
The ? is IIQ's placeholder — it injects the last aggregation timestamp automatically at runtime.

3 — How IIQ manages the timestamp internally
This is the key mechanic to understand:

First delta run  →  IIQ uses epoch (1970-01-01) or the stored "lastRefresh"
After each run   →  IIQ stores SYSTIMESTAMP into the application's "lastRefresh" attribute
Next delta run   →  IIQ injects that stored timestamp as the ? parameter
Think of it like a bookmark in a book — IIQ leaves the bookmark at the last page read, and next time starts from there.

You can inspect or reset it in the IIQ DB:

sql
-- Check the current lastRefresh stored by IIQ
SELECT name, attributes
FROM   sailpoint.spt_application
WHERE  name = 'OWN_AP12008';
Or reset it via the UI: Applications → ROBI → Edit → Advanced → Last Refresh (clear it to force a full re-delta from epoch).

4 — Interaction with your SCD2 procedure
This is important — the two processes must stay in sync:

SQL*Loader          →  loads LANDING table
SP_LOAD_STG_MASTER_ROBI  →  runs SCD2 MERGE, sets UPDATED_ON = SYSDATE on changed/new rows
IIQ delta task      →  runs AFTER the procedure, picks up rows WHERE UPDATED_ON >= lastRefresh
Your SP_LOAD_STG_MASTER_ROBI already sets UPDATED_ON = SYSDATE in both the MERGE UPDATE and INSERT branches — so no changes needed to the procedure. Every row touched by the SCD2 logic will be visible to the next delta run automatically.

5 — One edge case to handle: the race condition
If IIQ's delta task and SQL*Loader run close together in time, rows written during the IIQ delta window could be missed. The safe pattern is:

IIQ lastRefresh  =  T-1  (stored from previous run)
SCD2 proc runs   →  writes UPDATED_ON = T
IIQ delta runs   →  queries UPDATED_ON >= T-1  ✅ catches all of T
IIQ stores T+ε   →  next run starts from T+ε
To be safe, add a small buffer — subtract 5 minutes from the injected timestamp — by wrapping the application's delta query in a view that does the subtraction, or by scheduling the IIQ delta task to run at least 10 minutes after the SQL*Loader job completes.

3. IIQ Delta Task  → aggregate WHERE IS_CURRENT='Y' AND UPDATED_ON >= lastRefresh
4. IIQ stores new lastRefresh automatically

The analogy shift
Previously delta aggregation was a change-of-address notification. Now it's more like a full personnel file update — IIQ receives every record that was touched by the SCD2 run, whether the person got a new role (new current row) or left the building (expired row). IIQ's own correlation engine then decides what to do with each one.

1 — Updated delta aggregation query
Remove the IS_CURRENT = 'Y' filter. Add IS_CURRENT as a returned column so IIQ's rules can act on it.

sql
SELECT
    ROBID,
    FLID,
    REFOG,
    ENTITYID,
    ENTITY,
    MANAGER,
    LASTNAME,
    FIRSTNAME,
    COUNTRY,
    LANGUAGE,
    FUNCTION,
    EMAIL,
    TYPE,
    STATUS,
    ISTEST,
    ISRPA,
    IS_CURRENT,
    VALID_FROM,
    VALID_TO,
    SOURCE,
    UPDATED_ON
FROM OWN_AP12008.STG_MASTER_ROBI
WHERE UPDATED_ON >= ?
No IS_CURRENT filter — the ? timestamp alone drives what gets picked up.

2 — Why IS_CURRENT, VALID_FROM, VALID_TO must be in the schema
IIQ needs to see these columns as account attributes on the link, otherwise they're invisible to rules and policies. Make sure they are declared in the application schema:

xml
<Schema objectType="account" ...>
  <AttributeDefinition name="IS_CURRENT"  type="string"/>
  <AttributeDefinition name="VALID_FROM"  type="date"/>
  <AttributeDefinition name="VALID_TO"    type="date"/>
  ...
</Schema>
Think of the schema like a customs declaration form — only columns declared on the form get through into IIQ's data model. Undeclared columns are silently dropped even if the SQL returns them.

3 — One important problem: multiple rows per ROBID
Since STG_MASTER_ROBI is now SCD2, a single ROBID can have multiple rows — one current (IS_CURRENT='Y') and one or more expired (IS_CURRENT='N'). The delta query above could return both in the same run.

IIQ's JDBC connector expects one row per account. If it sees two rows with the same correlation key (ROBID or REFOG), behavior is undefined — it may create duplicate links.

The fix — deduplicate in the query, prioritize current row
sql
SELECT
    ROBID,
    FLID,
    REFOG,
    ENTITYID,
    ENTITY,
    MANAGER,
    LASTNAME,
    FIRSTNAME,
    COUNTRY,
    LANGUAGE,
    FUNCTION,
    EMAIL,
    TYPE,
    STATUS,
    ISTEST,
    ISRPA,
    IS_CURRENT,
    VALID_FROM,
    VALID_TO,
    SOURCE,
    UPDATED_ON
FROM (
    SELECT
        t.*,
        ROW_NUMBER() OVER (
            PARTITION BY ROBID
            ORDER BY IS_CURRENT DESC,   -- 'Y' sorts before 'N'
                     VALID_FROM DESC    -- most recent version wins
        ) AS RN
    FROM OWN_AP12008.STG_MASTER_ROBI t
    WHERE UPDATED_ON >= ?
)
WHERE RN = 1
Think of ROW_NUMBER() here like a tiebreaker rule in a competition — if a ROBID has both a current and an expired row touched in the same SCD2 run, the current row wins. If only the expired row was touched (departed identity), it wins by default.

5 — Letting IIQ decide what to do with IS_CURRENT = 'N'
Since you chose "IIQ correlation rules decide", the natural place to act is a Customization Rule on the application — it fires after each account is aggregated and can inspect IS_CURRENT:

java
// Customization Rule on OWN_AP12008
import sailpoint.object.*;

String isCurrent = (String) object.getAttribute("IS_CURRENT");

if ("N".equals(isCurrent)) {
    // Signal to IIQ that this account is inactive
    object.setAttribute("IIQDisabled", true);
}

return object;
Think of the Customization Rule as a processing clerk at the border — every aggregated account passes through it, and the clerk stamps "disabled" on the file if IS_CURRENT is N, before it enters IIQ's identity cube.
